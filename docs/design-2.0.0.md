# 📘 每日技术问答 v2.0.0 设计文档

**版本**: v2.0.0（规划中）
**核心主题**: 互动答题 & 成长系统

---

## 🎯 版本概述

v2.0.0 的核心目标是将插件从"被动推送"升级为"主动参与"——用户不再只是接收题目，而是可以作答、获得反馈、积累成长。

核心设计原则：
- **答题门槛低**：`/a <id> <回答>` 即可参与，无需额外配置
- **作弊无意义**：答案在回答前不可见，但我们不假设用户会作弊——目的是让大家学习记住，而不是严格考核
- **LLM 用在刀刃上**：只在用户主动回答时调用，判定是否认真作答

## 📊 功能范围

### 新增功能

| 功能 | 说明 |
|------|------|
| **互动答题** | 用户通过 `/a <id> <回答>` 主动作答 |
| **LLM 智能评审** | 判定是否认真作答 + 检测 AI 复制粘贴 + 评估覆盖的知识点 |
| **成长系统** | EXP（认真回答即得）+ Score（按知识点拆分，答满为止）|

| **榜单系统** | 周榜（`/lrank`）+ 个人成绩（`/myscore`）|

## 🔑 核心机制

### 1. 互动答题流程

```
用户发送: /a 42 HashMap 底层是数组+链表，JDK8 后引入红黑树...
                    ↓
LLM 判定（三合一）:
  ① AI 复制粘贴检测 → 若判定为直接复制 AI 输出 → 提示"请用自己的话作答"，拒绝
  ② 有效性检测     → 是否在认真回答（非刷关键词/无关内容）→ 无效则拒绝
  ③ 知识点覆盖评估 → 对照 default_ans，本次回答新覆盖了哪些知识点，折算成分数
                    ↓
系统检查①: 查询该题今天推送轮已累计得分
  - 已满分 → 不发 Score
  - 未满     → 发放本次覆盖的新增 Score（不超过剩余上限）
                    ↓
奖励发放:
  - 基础EXP: 只要是当天该题的【第一条有效回答】，就发放由系统设定的基础 EXP，后续再继续抢分补充答案将不再重复发放基础 EXP，防止刷屏刷经验。
  - Score: 本次新覆盖知识点对应的分值（当该题该群累计达到满分后不再发新增Score）。
  - 若用户命中了有效知识点，无论有没有拿到分，若本题依然还有剩余未拆解的 point 被遗漏，系统会自动提醒：“想看提示请发送 /h <id>”。
                    ↓
解锁答案: 用户可使用 /ans <id> 查看参考答案
```

### 2. Score 知识点分配机制

每道题满分 **10 分**，知识点和分值由题目维护者在导入题库时预先标注，存入 `problems.score_points` 字段（JSON 格式）。

LLM 不需要“猜”知识点，直接依据预设评分表检查覆盖情况。

**`score_points` 格式（JSON 存入字段）**：

```json
[
  {"idx": 0, "point": "底层数据结构（数组+链表）", "hint": "", "score": 2},
  {"idx": 1, "point": "JDK8 红黑树优化及阈值（8）", "hint": "", "score": 2},
  {"idx": 2, "point": "扩容机制（负载因子 0.75）", "hint": "", "score": 3},
  {"idx": 3, "point": "线程不安全原因", "hint": "", "score": 3}
]
```

> `idx` 显式存储索引，不依赖数组位置推断，后续重新排序也不会漂移。

**示例**（题目：HashMap 底层原理）：

```
第一位回答者: 覆盖 index 0+1 → +4 Score
第二位回答者: 新覆盖 index 2     → +3 Score
第三位回答者: 新覆盖 index 3     → +3 Score（满分，后续不再发 Score）
第四位之后:   不管回答什么       → 只得 EXP
```

**`problem_score_log.covered_mask`** 存储的是已覆盖索引的位掉码（`INTEGER`）：第 i 位为 1 表示 index i 已覆盖。新回答进来做位运算即可。

**如果题目没有填写 `score_points`**：LLM 降级为直接打 0-满分 综合分，不影响正常流程。

### 3. EXP 与 Score 的区别

| | EXP | Score |
|---|---|---|
| **核心定位** | 鼓励“勇于开口”。只要认真答了哪怕只沾边，也能拿到的参与安慰奖。 | 衡量“技术深度”。大家群策群力瓜分的硬核积分池。 |
| **获得条件** | 在全局冷却期内（默认 30 天，可配置）对该题的【首次】有效回答即可获得该领域配置的基础经验（`domain.base_exp`，默认 5）。冷却期内该题多次回答不再重复给基础 EXP，防止刷经验。 | 提交的回答成功命中了题库标准上**尚未被自己或全群其他人抢走**的知识点即可获得对应分值。 |
| **重置限制** | 经验值永远不重置，随账号终身累加。 | Score 可能对应具体的赛季（后续拓展），每道题目的池子随着新一轮推送而刷新重置。 |
| **上限瓶颈** | 每道题每人经过配置的一定天数之后才能再次获得 1 次基础 EXP 的机会。 | 只要该题的满分没被掏空，你可以连发 10 条消息把 10 分全榨干。 |

### 4. 答案保护与提示机制

#### 查看答案的条件
为了保证大家依靠 `hint` 抢分时的探索积极性，不能随时调用 `/ans <id>` 看答案，只有**满足以下至少一个条件**时才允许查看：
1. **全群已抢满**：本轮推送中该题的进度已达满分 (`is_complete = 1`)。
2. **时效已过**：当此题目到了第二天（或者经过了一次每日推送出新题的轮替）。
3. **特权调用**：请求者是机器人管理员。

*未满足条件时调用*：提示「⚠️ 本题还剩 X 分未被发掘，多看看提示再抢答一波吧！（或者等下轮出新题后再来查看答案~）」

#### 答案展示形态
一旦允许查看答案，展示的内容不止是纯文本，将同时公布官方考核点供大家学习核对：
```
📋 [题目 ID 42]
【参考答案原文】
HashMap 底层是...

【评分点分布】
🎯 1. 底层数据结构 (数组+链表): x 分
🎯 2. JDK8 红黑树优化及阈值: y 分
...
```

#### 下一考点提示 (Hint) 机制
为了促进“抢分大逃杀”的可玩性：
* 若当前的解答未将该题目的满分彻底刮分完毕，当任意玩家试图通过有效的方法作答时，系统会提示此题还有剩余分数未被挖掘。
* 此时用户可以主动发送 `/h <id>` 探寻答案线索。
* 系统会向目标题目对应的未解锁的剩余知识点池 (`score_points` -> `hint`) 中，**按逻辑排序顺次抛出**第一条尚未被大家打出的考点提示（就像剥洋葱一样）反馈给当前用户。
* 由于知识点天然具有前置依赖属性（逻辑顺序或时间顺序），这种顺次求出下一考点线索的方式，能一步一步引导大家拼凑线索、根据线索去检索资料，也是一种很好的查漏补缺学习模式。

### 5. LLM 判定设计

**触发时机**：仅在用户执行 `/a` 时调用一次，不监听全群消息。

**两种模式**：
- **情况 A**：题目有 `score_points` → 精确模式，返回 `covered_indices`（整数列表）
- **情况 B**：无 `score_points` → 降级模式，返回 `points_covered`（0~10 综合分）

**三合一判定策略**：

1. **AI 复制检测**（宽松，只拦明显粘贴）
   - 拦截：Markdown 多级标题（`##/###`）+ 内容完整无口语，两者同时具备
   - 放行：有口语、不完整、普通列表、白话重述，任意一项即可
   - 原则：宁可放行一个复制者，不冤枉一个正常作答的人

2. **有效性判断**
   - 无效：纯关键词堆砌（不成句）、与题无关、纯表情/感叹
   - 有效：口语化描述、残缺句、碎片回忆均算有效

3. **评分表覆盖**（仅情况 A）
   - 逐项判断是否覆盖，措辞不完全一致但理解正确也得分
   - 返回已覆盖的知识点 `idx` 整数列表

> 具体 Prompt 见 `llm/prompts.py`，填充逻辑见 `llm/judge.py`

**系统处理逻辑**：
- `ai_copied = true` → 拒绝
- `valid = false` → 拒绝，不发 EXP
- 有 `covered_indices` → 转成 `llm_mask`，`新增 = llm_mask & ~covered_mask`，只给新增索引对应的分值发 Score；写回 `covered_mask |= llm_mask`
- 有 `points_covered`（降级）→ 直接与已累计分取差，超出截断

## 🗄️ 数据库设计

### 激活现有字段

`users` 表中已有但未启用的字段：

| 字段 | 用途 |
|------|------|
| `exp` | 累计 EXP |
| `weekly_exp` | 本周 EXP（每周重置） |
| `score` / `total_score` | 累计 Score |
| `weekly_score` | 本周 Score（每周重置） |

### 新增表

#### `user_answer_log` - 用户答题记录

```sql
CREATE TABLE IF NOT EXISTS user_answer_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_qq TEXT NOT NULL,
    problem_id INTEGER NOT NULL,
    group_qq TEXT NOT NULL,
    answer_text TEXT NOT NULL,
    is_valid INTEGER DEFAULT 0,         -- LLM 判定是否有效
    ai_copied INTEGER DEFAULT 0,        -- LLM 判定是否 AI 复制
    covered_mask INTEGER DEFAULT 0,     -- 覆盖位掉码：第 i 位为 1 表示覆盖了 score_points[i] 知识点
    llm_feedback TEXT,                  -- LLM 点评
    exp_gained INTEGER DEFAULT 0,       -- 实际拿到多少基本经验（同人同天同题非首次为0）
    score_gained REAL DEFAULT 0,        -- 实际抢夺发放的 Score
    answer_date TEXT NOT NULL,          -- 回答日期 YYYY-MM-DD
    answered_at DATETIME DEFAULT (datetime('now'))
    -- （注：v2 中已移除 UNIQUE 约束，允许同一天内多次回答抢不同知识点的 Score）
);
```

#### `problem_score_log` - 题目每轮推送的得分进度

```sql
CREATE TABLE IF NOT EXISTS problem_score_log (
    problem_id INTEGER NOT NULL,
    group_qq TEXT NOT NULL,
    push_date TEXT NOT NULL,            -- 本次推送日期 YYYY-MM-DD
    total_score REAL DEFAULT 0,         -- 本轮已累计发出的总 Score（0-10）
    covered_mask INTEGER DEFAULT 0,     -- 所有回答者已覆盖知识点的并集掩码
    is_complete INTEGER DEFAULT 0,      -- 本轮是否已满分被榨干
    PRIMARY KEY (problem_id, group_qq, push_date)
);
```

**周期设计**：
- 同一题目每次被推送时，`push_date` 取当天日期，上轮满分不影响本轮累积
- `user_answer_log` 的 `answer_date` 也取当天，两者对齐，确保每 VBO 每天仅得一次 EXP

## 🎯 未来迭代计划 (Backlog)

### 1. 防摆烂触发机制
- **说明**：在群聊中监听特定“摆烂”关键词（如“不想卷了”、“开摆”）。
- **机制**：由 LLM 识别负面情绪，随机触发一次“技术小挑战”，成功完成后给予相应奖励。

### 2. 面试复盘系统 (Debrief)
- **指令**：`/debrief <复盘内容>`
- **存储**：`interview_debrief_log` 表（id, user_qq, group_qq, content, is_valid, llm_feedback, exp_gained, submitted_at）。
- **流程**：
  - 用户提交真实的面试题目和个人总结。
  - LLM 评估内容的真实性和技术价值。
  - **奖励**：每次有效复盘增加相应 EXP，每日限额 1 次以防刷分。
- **价值**：建立群内的技术沉淀，形成良好的分享氛围。

### 3. 多媒体与长内容展示优化
- **长内容转发**：在非 T2I 模式下，对于超长文本优先使用 AstrBot 的合并转发功能（Node Message）以减少群内刷屏。若环境不支持合并转发，则 fallback 为普通消息发送。
- **T2I 渲染优化**：
  - 在开启 T2I 模式时，将纯文本内容渲染为精美图片发送。
  - **处理逻辑**：由于当前 T2I 引擎暂不支持渲染内嵌图片，若内容包含图片组件，则自动 fallback 为“长内容转发”模式下的处理方法，确保图文混排内容的完整可见性。
